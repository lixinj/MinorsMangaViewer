我浏览了整个项目，结合最近的迭代，从代码质量、性能、UX、可维护性几个维度整理出以下可优化点。这些不是必须改的 bug，而是可以让项目更稳定、更好维护的方向。

1. 架构与状态管理
1.1 ReaderView 过于庞大
ReaderView.swift 目前已经接近 400 行，内部混合了：

阅读器主界面
toolbar 布局
自定义版本选择器
键盘/鼠标事件处理
双页渲染逻辑
建议：把版本选择器抽出为独立 VersionPicker 组件，把键盘事件处理交给专门的 ReaderKeyboardHandler 或 ViewModel 扩展。这样 ReaderView 只负责组合，测试和后续调整都更容易。

1.2 LibraryViewModel 职责渐多
LibraryViewModel 同时管理：

库加载
作者/子文件夹选择
搜索过滤
作品排序
文件名过滤
separateAnthology 状态
建议：把搜索过滤、排序逻辑抽到独立的小型结构（如 WorkFilter / WorkSorter），方便单元测试。

1.3 AppState 与 LibraryViewModel 的数据边界
AppState 只保存 libraryURL，而 LibraryViewModel 负责加载。这本身没问题，但 MainWindow 同时持有 appState 和 libraryViewModel，且 SidebarView 通过 @EnvironmentObject 访问。当项目变大时，这种全局可访问的 ViewModel 会让数据流变模糊。

建议：考虑把"当前选中的路径上下文"封装成一个不可变的 LibraryContext 对象，由 AppState 持有，ViewModel 接收它工作。

2. 性能
2.1 缩略图生成在主线程
ThumbnailCache.swift 整体标记了 @MainActor，且 thumbnail(for:) 虽然声明为 async，但内部 NSImage(contentsOf:)、resized(toMax:)、jpegData 都在主线程执行。当作品较多或图片较大时，网格滚动会卡顿。

建议：

把图片加载和缩放移到后台 Task.detached 或自定义 DispatchQueue
主线程只负责最终的 NSImage 赋值
磁盘写入也可以异步到后台队列
2.2 阅读器图片没有后台加载
ReaderViewModel.swift 中 PageImage 使用 NSImage(contentsOf:) 同步加载，且 setCachedImage 在主线程。单张大图可能造成界面卡顿。

建议：图片加载和缓存改用后台任务，主线程只更新 UI。

2.3 双页合成图方案已废弃，但代码残留
之前尝试过的 combineImages 函数已经删除，这是好事。但建议保留手动计算 frame 的注释，说明为什么不用 .scaledToFit()，方便后续维护。

3. 健壮性与错误处理
3.1 FolderScanner 错误处理偏轻
FolderScanner.swift 很多地方用了 try?，失败时静默返回空数组。用户遇到"某些作品不显示"时很难定位。

建议：至少把扫描失败的目录路径打印到日志，或考虑暴露一个"上次扫描错误"状态给 UI。

3.2 ReadingProgressStore 键值可能冲突
ReadingProgressStore.swift 用 path.path 作为 UserDefaults 的 key。如果用户把库移动到不同路径，进度会丢失；不同作品如果路径相同（不同机器）会冲突。

建议：用作品/版本的稳定标识（如文件夹名哈希 + 作者名）作为 key，而不是绝对路径。

3.3 SecurityBookmarkStore 没有处理 stale bookmark
AppState.swift 加载 bookmark 时虽然检查 isStale，但发现 stale 后只是打印，没有重新保存。长期可能失效。

建议：stale 时重新创建并保存 bookmark。

4. UI/UX
4.1 版本选择器的菜单位置硬编码
当前版本菜单使用 padding(.top, 50) 和 padding(.leading, 150) 硬编码定位。窗口大小变化或字体变化时可能错位。

建议：用 GeometryReader 读取版本按钮的实际 frame，再计算菜单位置，或者使用 SwiftUI 原生的 .popover/.overlay 锚定到按钮。

4.2 键盘监听分布两处
ReaderView 同时用 .onKeyPress 和 NSEvent.addLocalMonitorForEvents，SidebarView/FinderLikeGridView 也用各自的 monitor。当功能继续增加时，容易出现按键冲突或重复消费。

建议：考虑统一到一个键盘路由层，按当前"焦点区域"（侧边栏、网格、阅读器）分发事件。

4.3 ReaderView 的 toolbar 与 ZStack overlay 耦合
版本菜单通过外层 ZStack 的 overlay 实现，点击菜单外部不会自动关闭（除非点击翻页区域或按键）。这与 macOS 原生下拉菜单行为不同。

建议：可以考虑用 .popover 或 Menu 组件，它们会自动处理点击外部关闭。如果坚持自定义，添加一个全屏透明点击层在菜单展开时捕获外部点击。

4.4 主界面路径条可能过长
MainWindow.swift 的 pathView 显示完整路径，如果作者名/子文件夹名很长，会挤压 toolbar 按钮。

建议：给路径文本加 lineLimit(1) 和 truncationMode(.middle)，或限制最大宽度。

5. 可测试性
5.1 业务逻辑集中在 ViewModel/View 中
目前只有 NamingParserTests.swift 有单元测试。FolderScanner、ReadingProgressStore、版本标签解析等核心逻辑都缺少测试。

建议：

为 WorkVersion.displayLabel 添加测试
为 FolderScanner.collections(for:) 创建临时目录测试
为 ReadingProgressStore 抽象一个存储接口，便于测试
5.2 ReaderViewModel 依赖全局单例
ReadingProgressStore.shared 是全局单例，ReaderViewModel 直接引用，测试时难以 mock。

建议：通过构造函数注入，默认值仍可用 shared。

6. 代码风格与细节
6.1 魔法数字较多
ReaderView.swift 中大量硬编码的 padding、width、offset（如 150、280、50、120 等），没有命名常量。

建议：提取为 private enum LayoutMetrics 或 private extension CGFloat，提升可读性。

6.2 注释可以更精确
部分函数已有中文注释，但一些复杂的 UI 计算（如双页 frame 计算）缺少注释。

建议：在关键算法处补充"为什么这样做"的注释。

6.3 PageImage 与双页逻辑重复加载图片
PageImage 和 doublePageContent 都从 cache 取图，但 fallback 路径不同。可以考虑统一图片获取逻辑。

优先级建议
如果只能选几个先做，我建议：

高优先级：把 ThumbnailCache 和 ReaderViewModel 的图片加载移到后台线程，这是目前最影响体验的。
中优先级：抽出独立的 VersionPicker 组件，并改用几何定位或原生 popover。
中优先级：为 FolderScanner 和 WorkVersion.displayLabel 补充单元测试。
低优先级：统一键盘事件路由，清理魔法数字。
